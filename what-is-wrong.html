What is wrong with the inaccessible page?

No skip link

A skip link helps users using assistive technology to skip repeated content such as headers and get straight to the main content of the page.

Without the skip link, it can be frustrating for users to repeatedly hear the same boilerplate information.

Last name input doesn't have a label

While it looks like there's a label for the last name input, it has not been correctly coded to link to the input.

This means clicking the text does not focus on the input. When the input is focussed the text is not read out on a screen reader.

NHS number doesn't have a link to the hint text

Hint text can be helpful and we can use aria tags to link it to an input.

By using 'aria-describedby="hint-text-id"' the hint text will be announced to screen reader users when the focus is on the input.

Autocomplete off

Browsers autocomplete function is helpful as it allows common information to be input quickly.

By restricting this or marking up the input incorrectly it can be frustrating to our users who now have to input that data manually.

Tab order

When using a keyboard, tabbing through the page takes you in an illogical order. This could be hard to follow for users. It's a much better experience to follow the logical flow of the page.

No focus styles on the first name input

There's no indication that the first name input is actually focused. Focus styles are important for showing where on the page the user currently is.

Buttons are too close together

The 'next' and 'cancel' buttons are too close together. A user with Parkinsons could click 'cancel' by mistake and have to spend time entering their information again.

Email and NHS Number types are wrong

Having the correct semantic information for inputs means that the correct keyboard displays for users on mobiles.

This is beneficial for all users as a number pad will appear for numbers and @ symbol may be more available for emails.

Information under the buttons could be missed

A user using a screen reader may not be aware of the content under the buttons. If they fill in their information and click next, they might not know they've missed any information at all.

Elements not contained by landmarks

Writing our pages in semantic and correct HTML5 enables the page to be parsed correctly by assistive technologies.

In this instance, there is no main tag on the page. Content could be missed if landmark tags are incorrectly labelled.

Next button instead of Continue

Continue is the standard text for a button that progresses a user in a journey.

'Next' is a common command for speech recognition users. Users saying 'click next' might not be able to access the 'Next' button.

Assistive technology may interrupt 'Next' as the 'next bit of content, 'next link' or 'next tab'. Using Continue makes it easier for speech recognition software to target the button.
